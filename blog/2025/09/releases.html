Releases

Let's look through the change logs!

Two Releases Before May

I have blog post drafts for these releases, but didn't get around to publishing them.

  • Oils 0.28.0 - March 16th - changelog

    I mentioned Koichi Murase's excellent work in the Oils 0.27.0 announcement, and there was even more in this release. He continued the overhaul of bash-style arrays in OSH, and fixed many bugs along the way.

    I appreciate when contributors go above and beyond, making the project better for everyone!

  • Oils 0.29.0 - May 2nd - changelog

    Matthew Davidson landed the initial implementation of bind -x, a commonly used interactive shell feature. Samuel Hierholzer added io.captureAll(), which captures stdout and stderr simultaneously.

Four Releases in a Week in June

These four releases were motivated by performance! That wasn't planned, but I like to keep track of benchmarks on the release quality page when there are notable changes.

  • Oils 0.30.0 - June 20th - changelog

    Thanks to Adam Bannister and Matthew Davidson for adding spec tests.

    I implemented features for ble.sh, including the "private" builtin sleep, and the ability to print source code with declare -f.

    I fixed a subtle bug with respect to $PWD, which Samuel found. It deserves its own "bug post-mortem", but for now you can see it mentioned it at the end of this comment:

This post describes an interesting bash bug, unrelated to OSH.

  • Oils 0.31.0 - June 21st - changelog

    I reverted the implementation of declare -f, because it caused a performance regression! I re-implemented it with a simpler data structure.

  • Oils 0.32.0 - June 24th - changelog

    The regression created a "bug up my ass" regarding performance. For years, we've been tracking an autotools configure workload, on which we are slower than bash.

    So I profiled the workload, and came up with the solution of "private" builtins rm and cat, controlled by shopt --set rewrite_extern. This (sound) optimization makes OSH faster than bash on autotools and other I/O-bound workloads! I will write more about this.

  • Oils 0.33.0 - June 25th - changelog

    I was encouraged by the progress, so I also turned on the Souffle GC optimizations that Melvin Walls implemented last year. This makes OSH faster than bash on compute-bound workloads.

    Unfortunately, we later reverted this because of a bug. We should write about this after it's re-enabled.

Kicking Off the 4th Grant

  • Oils 0.34.0 - July 17th - changelog

    James Chen-Smith landed the initial implementation of fc -l.

    Will Clardy improved args.ysh to respect --.

    Koichi Murase fixed the printing of shell variables, multiple word splitting bugs, and fixed an adjacent bug in the read builtin.

  • Oils 0.35.0 - August 30th - changelog

    I did a ton of work on regtest/aports, a batch job that builds all Alpine Linux packages in aports/main with OSH. More on this in the next post.

    Aidan Olsen fixed the evaluation order of redirects in simple commands — a great bug that Samuel found with the oily-pine test harness. (The fourth grant, including regtest/aports, was inspired by that work.)

    Koichi Murase did an amazing overhaul of our word splitting algorithm, fixing 5 or more bugs in the process, including the most common one surfaced by regtest/aports. This was essential!

Notable Improvements

I focused on crediting contributors above, because I wanted to keep this post short. But now it feels incomplete!

YSH

Let me at least remind myself of all the YSH improvements:

  • Improved the "debug stack" for stack traces, prompted by a user question on Zulip
    • e.g. io->evalExpr() creates an entry in the stack
  • Some work on Hay.
    • This is probably the biggest open design issue in YSH, which Samuel has been helping with.
    • Implemented io->evalInFrame(), and then replaced it with a more orthogonal io->eval(in_captured_frame=true)
  • Added the --eval and --eval-pure flags
    • Work on a pure interpeter, for both pure functions and --eval-pure
  • Added dynamic control flow builtins (break continue ...)
    • The rule could be that YSH has static control flow, while OSH has dynamic control flow
  • First pass of quote.ysh in the stdlib
    • Motivated by a question from Julian Brown. I want to get back to this.
  • shopt --set parse_ysh_string disallows --flag=r'foo'
    • I noticed this YSH quirk when writing the Vim highlighter for YSH: oils.vim
  • Added the invoke builtin, to generalize the non-orthogonal builtin and command
    • invoke --show is also better than type -a
    • private builtin sleep, mentioned above
    • private builtin rm and cat, mentioned above
  • Added Proc.docComment()
    • We still need __sh_function__, which I believe replaced getShFunction()
    • We probably need a notion of read-only Obj instances, so that mutating __sh_function__ can't corrupt the interpreter state
  • Disallow test foo in favor of test -n foo
    • based on feedback from bar-g
  • Make Unicode in Eggex consistent with the rest of YSH
    • \y01 is now the syntax for bytes, not \x01
    • thanks again to bar-g for feedback
  • Fixed many bugs reported by users

OSH

Selected changes:

  • Fixed cross-process tracing with export SHELLOPTS
  • In shell, the read builtin omits NUL bytes in the input, which surprised me. OSH now does this.
  • I noticed that shell doesn't have a good idiom for running multiple processes in parallel, while respecting errexit
    • #blog-ideas > How to run processes in parallel, and check for errors
    • I fixed bugs in this area, so OSH is more compatible

Common Infrastructure

  • Got a minimal bin/hello working under Win32
    • I've come around to the fact that a Windows build is useful, though we'll need contributions here
  • Experimented with a bytecode-like representation, to fix the "break as C++ exception" problem
  • Added a script to create a statically linked oils-for-unix
    • We're using this with musl libc, and even glibc
  • Various build and C++ portability fixes, motivated by user reports

Docs

  • Document mixing of OSH and YSH code, a common FAQ
  • Document buffered versus unbuffered I/O

Conclusion

OK, now I feel reasonably caught up on the last 6 months! I briefly reviewed 8 releases, with a focus on crediting contributors.

Then I summarized notable changes in OSH, YSH, common infrastructure, and docs.

Upon reflection, it does seem like there weren't many big features in the last 6 months. The biggest YSH feature was probably oils.vim, the Vim syntax highlighter. (I wrote about Three Algorithms for YSH Syntax Highlighting, including coarse parsing.)

But there are good signs of progress in the next post: Links to Explain Oils in 2025.

Feel free to ask questions in the comments!

Appendix: Metrics for the 0.35.0 Release

Let's review release metrics, which help me keep track of the project. The last review was in May, in Metrics for the 0.29.0 Release.

Docs

We continue to improve the Oils Reference. In addition to adding new topics, I polished existing topics. Thanks to bar-g for feedback on the docs.

  • Doc Metrics for 0.29.0 - 428 topics with first pass, 455 marked implemented, 496 unique
  • Doc Metrics for 0.35.0 - 442 topics with first pass, 466 marked implemented, 510 unique

Spec Tests

OSH made good progress, with 91 new passing spec tests.

  • OSH spec tests for 0.29.0: 2517 tests, 2248 passing, 134 failing
  • OSH spec tests for 0.35.0: 2608 tests, 2328 passing, 156 failing

But it also feels like there are too many failing tests. (Many of them are related to the missing GLOBIGNORE feature.) We're now prioritizing bug fixes with regtest/aports.

Let's look at the C++ translation:

  • C++ spec tests for 0.29.0 - 2240 of 2248 passing - delta 8
  • C++ spec tests for 0.35.0 - 2322 of 2328 passing - delta 6

The delta is due to the bind builtin, which we're working on in Python first. There's also a new anomaly related to new history spec tests, which I want to look into.

YSH has 40 more tests passing:

  • YSH spec tests for 0.29.0: 1090 tests, 1022 passing, 68 failing
  • YSH spec tests for 0.35.0: 1133 tests, 1062 passing, 71 failing

They all pass in the C++ translation:

  • YSH C++ spec tests for 0.29.0: 1020 of 1022 passing, delta 2
  • YSH C++ spec tests for 0.35.0: 1060 of 1062 passing, delta 2

(The delta is again due to a Dict ordering issue in Hay, which is harmless right now)

Benchmarks

No significant changes in parser speed:

  • Parser Performance for 0.29.0: 12.9 thousand irefs per line
  • Parser Performance for 0.35.0: 12.3 thousand irefs per line

Or memory usage:

  • benchmarks/gc for 0.29.0: parse.configure-coreutils 1.65 M objects comprising 41.0 MB, max RSS 46.6 MB
  • benchmarks/gc for 0.35.0: parse.configure-coreutils 1.65 M objects comprising 44.6 MB, max RSS 50.8 MB

Runtime

A compute-bound workload is the same speed:

  • benchmarks/gc-cachegrind for 0.29.0 - fib takes 25.4 million irefs, mut+alloc+free+gc
  • benchmarks/gc-cachegrind for 0.35.0 - fib takes 25.4 million irefs, mut+alloc+free+gc

This is the infamous autotools configure workload, which unfortunately has noisy measurements:

  • Runtime Performance for 0.29.0 on 2 machines
    • 0.96x and 1.05x the speed of bash on configure.cpython
    • 1.04x and 1.10x the speed of bash on configure.util-linux
  • Runtime Performance for 0.35.0 on 2 machines
    • 0.95x and 0.87x the speed of bash on configure.cpython
    • 1.04x and 1.00x the speed of bash on configure.util-linux

It improved due to the rewrite_extern optimization, but it's not super clear on this particular run. It's more clear in the previous release:

  • Runtime Performance for 0.34.0 on 2 machines
    • 0.96x and 0.93x the speed of bash on configure.cpython
    • 0.96x and 0.93x the speed of bash on configure.util-linux

So here, we're beating bash on 2 workloads, measured on 2 different machines. (The statically-linked oils-for-unix beats bash by an even larger margin.)

At first I thought that the 0.35.0 release could be slower because I temporarily disabled the Souffle GC rooting optimizations. But, now that I look, that happened in 0.34.0. So I think it's just benchmark noise.

Code Size

Even though many new tests pass, our code is still short:

  • cloc for 0.29.0
    • 25,185 significant lines in OSH, 6,380 in YSH, 1,691 in data languages
    • 6,112 lines of hand-written C++
  • cloc for 0.35.0
    • 25,594 significant lines in OSH, 6,617 in YSH, 1,690 in data languages
    • 6,401 lines of hand-written C++

The generated C++ is proportional:

  • oils-cpp for 0.29.0 - 128,449 physical lines
  • oils-cpp for 0.35.0 - 193,303 - 62,484 = 130,819 physical lines (accounting another source file)

And compiled binary size is proportional:

  • ovm-build for 0.29.0: 2.40 MB of native code (hoover, under GCC, on Debian 12)
  • ovm-build for 0.35.0: 2.42 MB of native code (hoover, under GCC, on Debian 12)
