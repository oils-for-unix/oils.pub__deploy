Recap: Regular Languages and Doc Languages

  1. I finished setting up the new oils.pub domain. Thanks to everyone who pointed out broken links and oversights.
    • This involved a lot of faffing with HTML, of course.
  2. So I started working on an HTML5 subset called HTM8.
    • The idea is to express HTML declaratively, with regular languages.

(I named it "HTM8" because I consider it part of J8 Notation. But as I write this, I'm considering Regular HTML / RHTML, because it doesn't use the J8 string syntax, as JSON8 and TSV8 do.)

For context, here's a post that Claude AI found, showing how hard it is to write correct HTML processors: A History of HTML Parsing at Cloudflare: Part 1 (2019)

And I just noticed that they omitted the infamous CloudBleed vulnerability:

Cloudflare Reverse Proxies Are Dumping Uninitialized Memory (chromium.org via Hacker News)
3238 points, 1014 comments - on Feb 23, 2017

Parsing is still hard, and centralized architectures are fragile.

HTM8 / RHTML

I went through 3 iterations of the HTM8 design, aimed at simplicity and compatibility. One issue I remember getting tripped up on was:

<a href=foo/>
  • Is that a self-closing tag with href=foo, ending with />?
  • Or is it an opening tag with href=foo/ ?

It's the latter, because HTML5 doesn't have self-closing tags. But this example has implications for expressing the syntax with regular languages. It should be a single pass of parsing, not the "progressive" strategy inherited from our doctools/ library.

HTM8 isn't done, but it's close.

The last thing I did was write a YSH script to retrieve test data from https://commoncrawl.org/. I liked using YSH — it reduces shell/awk/Python cacophony — but there are also gaps. The next year will be about filling these gaps!

ul-table: Markdown Tables Without New Syntax

On the other hand, I did finish a related "doc languages" project. It even has documentation, which has been buried at the end of published docs:

  • ul-table: Markdown Tables Without New Syntax

We use Markdown everywhere, but it doesn't have tables. So we abuse unordered lists, in a way that ends up looking similar to tables in reStructuredText:

<table>

- thead
  - Shell
  - Version
- tr
  - [bash](https://www.gnu.org/software/bash/)
  - 5.2
- tr
  - [OSH](https://oils.pub/)
  - 0.29.0

</table>

ul-table detects the HTML corresponding to this structure, and transforms it into a table. The syntax above is valid Markdown, which means it "degrades" gracefully on sites like Github. You'll see bulleted lists, instead of a table.

I should write a real blog post about this. It was a success, in that I started using it immediately, all over our docs, and all over the blog.

For example, I ported a table-heavy post to ul-table: What Oils Looks Like in 2024. It looks the same, but the source is easier to write and maintain.

Under the Hood

OK, so regular languages and #doc-languages were one area of work.

Another area was "under the hood" cleanup, involving our metalanguages. I like cleaning up code after implementing features.

Zephyr ASDL

Zephyr ASDL is our DSL for algebraic data types, borrowed from CPython. I did an experiment toward "boxless" optimization, to eliminate GC objects:

  • #zephyr-asdl > Subtypes: CompoundWord < List[word_part]

So .asdl schema files support a new syntax for subtypes, and the toolchain generates typed Python and C++ code.

I used this feature in one place. It makes the code cleaner, but I'm not sure if it's a big win overall, so I didn't start using it everywhere. I think I used it one more time in the last few months.

mycpp

mycpp is our Python to C++ translator. It's always been the messiest part of the codebase, since it reaches into the internals of MyPy. I spent several days on a big cleanup:

  • #yaks > Big cleanup: mycpp has 5 separate passes

So it's now a pretty clean program, structured like a compiler. I still dislike the visitor pattern, but not as much. I can see its utility.

This involved refactoring the GC rooting optimizations that Melvin did last year, which use Souffle datalog. This feature makes a lot of sense, and I feel more confident about it now that the mycpp has been cleaned up,

On the other hand, we still have the anomaly shown by benchmarks/osh-runtime. For example, on CPython's configure, OSH makes around 5% fewer syscalls than bash, which is great. But it can still be around 5% slower, not faster. This bugs me, but we may have to let it be for now.

I'm also wondering if we can use Souffle datalog for other optimizations. In my limited understanding, it strength is that you can take a global view of the program, as opposed to a function-by-function view.

Pretty Printing

I think Wadler-style pretty-printing counts as a new metalanguage! You use it by generating a doc_t intermediate representation. Thank you to Justin Pombrio for implementing it.

We enabled pretty printing with the = operator in June 2024.

It's been working well, so I retired a different, crappy pretty printer I wrote many years ago for the osh -n debugging feature.

Instead, we generate the new Wadler-style IR. This fixed several bugs, which makes me happy. It uses more memory than I'd like, but I think it's OK because it's for debugging.

Much like Souffle datalog, I feel like I should sit down and understand this area more thoroughly. But there are so many other things to do!

async await and Event Loops

Confetti reminded me of catbrain, which I briefly mentioned in the Oils retrospective. It's a design exercise, for everything YSH doesn't have, but could have:

  • It might be about intra-process glue
    • e.g. with select() loops and C APIs
  • which is complementary to the inter-process glue of shell.

It also might work on Windows. I've come around to the idea that Windows is a problem that should be solved:

Everyone quotes command line arguments the wrong way (on Windows) (learn.microsoft.com via lobste.rs)
16 points, 14 comments on 2025-03-24

I don't have any concrete results now, but here's a list of research topics:

  • Coroutines and async await
    • I asked Claude to generate Python and JavaScript code, for comparison
    • And Lua code. Lua has has symmetric coroutines, in contrast to the asymmetric async await. ("Call" and "return" are considered asymmetric.)
    • And Tcl code. Tcl got coroutines in 2012. This is the same year I was playing with the Tornado async framework, and the same year that Guido started Tulip, the prototype for Python asyncio.
  • Event Loops are complementary to coroutines
    • For networking
    • And GUIs, on Windows / OS X / Linux
    • Event loops are under-appreciated glue!
  • Windows system calls (I abandoned Windows programming in 2007 or so.)
    • For starting processes and pipelines
    • As of 2018, Windows has Unix-style pseudo-terminals (PTYs)! A trillion dollars and a near monopoly can't overcome the inertia of the narrow waist.
    • Can FANOS work on Windows? We need something like Unix sockets and file descriptor passing.

Summary

  • Progress on doc languages: an HTML5 subset, and a nice way to create and maintain tables.
